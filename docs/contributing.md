# 贡献指南

感谢您有兴趣为 Minecraft 开服指南文档项目做贡献！我们欢迎所有形式的贡献，包括但不限于内容改进、错误修正、新章节添加等。

## 贡献方式

### 报告问题
如果您在阅读文档时发现任何错误、遗漏或不清楚的地方，可以通过以下方式报告：
1. 在 GitHub 上提交 Issue
2. 详细描述问题所在位置和具体内容
3. 如果可能，提供正确的信息或改进建议

### 内容贡献
如果您希望添加新内容或改进现有内容，请遵循以下流程：

1. **创建 Issue 讨论**
   在开始编写之前，先创建一个 Issue 来讨论您的想法，确保与项目目标一致。

2. **获得批准**
   维护团队会评估您的提议，并给出反馈意见。

3. **创建特性分支**
   ```
   git checkout -b feature/your-feature-name
   ```

4. **编写文档**
   - 遵循现有的文档风格和格式
   - 确保内容准确且经过实际验证
   - 添加必要的示例和截图

5. **本地预览验证**
   ```
   npm run docs:dev
   ```

6. **提交 Pull Request**
   包含清晰的变更说明和相关 Issue 链接。

## 内容质量标准

为了保证文档的质量，请在贡献时遵循以下标准：

### 准确性
- 所有技术内容必须经过实际测试验证
- 命令和配置示例需要标注适用平台（Windows/Linux）
- 性能建议应提供实际测试数据支撑

### 清晰度
- 使用简洁明了的语言表达
- 避免使用模糊表述，如"可能"、"大概"
- 优先使用"实测"、"建议"、"警告"等明确用语

### 完整性
- 新增指南需包含实际测试环境信息（操作系统/服务端版本）
- 配置示例必须标注适用版本范围
- 敏感操作（如内存设置）需添加警告提示

### 格式规范
- 使用 Markdown 格式编写
- 遵循统一的标题层级结构
- 适当使用列表、代码块、表格等元素提高可读性

## 技术写作建议

### 语言风格
- 面向中文用户，使用地道的中文表达
- 保持友好、专业的语气
- 避免过于口语化或过于学术化的表达

### 结构组织
- 每个页面应该有一个明确的主题
- 使用小节划分复杂内容
- 在开头提供简要概述
- 在结尾提供相关内容链接

### 示例代码
```

java -Xmx4G -jar server.jar
```
